## 遇到的问题

1. **Docker Desktop未运行**: 改用Qdrant本地嵌入式模式（QdrantClient(path=...)），与ChromaDB PersistentClient用法类似
2. **SparseVector格式错误**: dict不能直接作为PointStruct的稀疏向量，必须使用SparseVector(indices=[...], values=[...])对象
3. **Point ID必须是UUID**: Qdrant本地模式要求point ID必须是有效的UUID字符串，实现了_str_to_uuid()将任意ID转为UUID5
4. **向量命名不一致**: 初始稠密向量无名而稀疏向量名为"sparse"，导致upsert失败。修复为统一命名"dense"
5. **bge-large-zh-v1.5下载超时**: 网络不可用，回退到本地缓存的bge-small-zh-v1.5 (512维)
6. **UTF-8编码问题**: Windows环境GBK编码导致中文输出乱码，使用-X utf8=1解决
7. **稀疏向量本地模式不支持**: 当前Qdrant本地版本不支持稀疏向量，代码已自动降级为纯稠密检索

## 解决方案

1. 实现Qdrant双模式架构，优先支持本地嵌入式（零依赖）
2. 在所有add_documents路径中统一使用SparseVector对象格式
3. 使用uuid.uuid5(DNS namespace, id_string)生成确定性UUID
4. 集合创建时统一使用命名向量格式{"dense": VectorParams(...)}
5. 设置HF_HUB_OFFLINE=1强制使用本地缓存模型
6. Python脚本添加-X utf8=1参数
7. _init_collection中try/except自动检测稀疏向量支持，add_documents和hybrid_search根据_sparse_supported标志自适应

## 经验与改进建议

1. **Qdrant本地模式是绝佳的零依赖方案**: 无需Docker，与ChromaDB使用体验类似但性能更好
2. **UUID转换需要保持幂等性**: uuid.uuid5()可确保相同字符串始终生成相同UUID，便于更新和删除
3. **向量命名必须全局一致**: collection创建时、upsert时、query时三处必须保持相同的命名约定
4. **稀疏向量在当前Qdrant本地版本受限**: 后续升级到Docker版本可解锁，目前已做优雅降级
5. **模型下载应有离线备选**: 建议将bge-large-zh-v1.5提前下载到本地缓存，或增加模型下载的超时/重试机制
