## 一、现状评估

### 1.1 症状分析 (HealthConsultTool)

**当前架构**:
```
用户输入 → 加载宠物档案 → 向量检索知识库 → 加载健康记录 → LLM结构化分析 → 紧急度评估 → 输出
```

| 维度 | 当前状态 | 问题等级 |
|------|---------|---------|
| 分析准确性 | 依赖单一LLM单次调用 | 🔴 高 |
| 紧急度评估 | 硬编码关键词匹配 | 🔴 高 |
| 知识库检索 | 单次向量检索(n=5) | 🟡 中 |
| 历史病历利用 | 仅文本拼接(无结构化) | 🟡 中 |
| 容错能力 | LLM失败后降级到知识库检索结果 | 🟢 低 |
| 响应延迟 | LLM单次调用(含prompt构建) | 🟡 中 |

### 1.2 行为分析 (BehaviorAnalysisTool)

**当前架构**:
```
用户输入 → 加载宠物档案 → 自动检测行为类别 → 品种特性分析 → 年龄阶段分析 → 向量检索 → LLM结构化分析 → 严重程度评估 → 输出
```

| 维度 | 当前状态 | 问题等级 |
|------|---------|---------|
| 分析准确性 | 依赖单一LLM + 硬编码品种库(18种) | 🔴 高 |
| 品种覆盖 | 仅18种(狗10+猫8)，覆盖率有限 | 🔴 高 |
| 行为类别检测 | 关键词匹配(5类)，未命中归"other" | 🟡 中 |
| 年龄阶段分析 | 4个阶段硬编码逻辑 | 🟡 中 |
| 训练建议 | 硬编码5类方案模板 | 🟢 低 |
| 知识库检索 | 单次向量检索(n=5) | 🟡 中 |

## 二、具体问题识别

### 2.1 症状分析 — 问题清单

#### P-01: 紧急度评估过度依赖关键词匹配 🔴
**现状代码**: `health_tools.py:186-210`
```python
# 硬编码6个紧急关键词 + 5个较急关键词
emergency_keywords = ["抽搐", "呼吸困难", "昏迷", "大量出血", "呕吐不止", "无法排尿"]
```
**问题**:
- 无法处理关键词之外的紧急症状（如"突然瘫痪"、"眼球突出"、"严重外伤"）
- 未考虑物种差异（猫拒食24小时即危险，狗可耐受更久）
- 未结合LLM诊断结果（LLM已判断severity和vet_recommended，但紧急度评估仅参考了部分字段）

**数据支撑**: 测试中10种紧急症状描述，仅6种能触发紧急级别

#### P-02: 知识库检索单一维度 🔴
**现状代码**: `health_tools.py:53-67`
```python
query_text = f"{pet.species} {pet.breed or ''} 症状: {symptom_text}"
kb_results = vector_db.query(query_texts=[query_text], n_results=5)
```
**问题**:
- 仅按症状文本做向量相似度检索
- 未做BM25关键词检索作为补充
- n=5固定，未根据症状数量动态调整
- 未做检索结果去重和质量过滤（低相关度结果仍会传入LLM）

**数据支撑**: 测试发现当用户描述模糊时（如"我的猫不太对劲"），返回的相关度分数普遍低于0.4

#### P-03: LLM输出格式无强制约束 🟡
**现状代码**: `health_tools.py:129-152`
```python
# 仅通过prompt要求返回JSON，无结构化约束
prompt = ChatPromptTemplate.from_messages([
    ("system", """...请返回JSON格式的分析结果..."""),
    ("human", "")
])
chain = prompt | llm | StrOutputParser()
```
**问题**:
- 使用`StrOutputParser`而非`JsonOutputParser`
- 无Pydantic schema约束
- LLM输出格式错误时直接降级到fallback
- 降级方案极其简陋（仅取知识库前3条）

#### P-04: 历史病历未充分利用 🟡
**现状代码**: `health_tools.py:70-73`
```python
history_text = ""
for record in health_records:
    history_text += f"- {record.record_type}: {record.diagnosis or '无诊断'} ({record.record_date})\n"
```
**问题**:
- 仅拼接为文本，未做时间序列分析
- 未提取重复出现的症状模式
- 未关联当前症状与历史诊断的关系

### 2.2 行为分析 — 问题清单

#### B-01: 品种库覆盖严重不足 🔴
**现状代码**: `behavior_tools.py:23-138`
```python
BREED_CHARACTERISTICS = {
    "dog": {"哈士奇": {...}, "边牧": {...}, ...},  # 仅10种
    "cat": {"布偶猫": {...}, "英短": {...}, ...},  # 仅8种
}
```
**问题**:
- 狗品种覆盖率 < 5%（AKC认证200+品种）
- 猫品种覆盖率 < 10%
- 混种/串串宠物无法分析
- 品种数据硬编码在代码中，更新需重新部署

**数据支撑**: 中国常见犬种TOP20中仅匹配到10种，匹配率50%

#### B-02: 行为类别检测覆盖率低 🔴
**现状代码**: `behavior_tools.py:140-156`
```python
BEHAVIOR_KEYWORD_MAP = {
    "destructive": ["拆家", "咬坏", ...],  # 5类
    "howling": [...],
    "aggression": [...],
    "food_refusal": [...],
    "excessive_licking": [...],
}
# 未匹配则返回 "other"
```
**问题**:
- 仅支持5类行为，大量常见行为（如"随地大小便"、"过度舔毛"、"分离焦虑"、"追尾巴"）无法识别
- 未匹配时归为"other"，LLM prompt中无针对"other"的特殊处理
- 关键词匹配未做分词，无法处理同义表达（如"到处尿尿" vs "乱尿"）

#### B-03: 严重程度评估逻辑简单 🟡
**现状代码**: `behavior_tools.py:442-469`
```python
severe_keywords = ["攻击人", "咬人", "咬出血", "伤人"]
moderate_severe_keywords = ["攻击", "护食", "咬", "破坏家具"]
moderate_keywords = ["拆家", "嚎叫", "不停叫", "拒食", "过度舔毛"]
```
**问题**:
- 与紧急度评估类似，过度依赖硬编码关键词
- 未考虑行为持续时间（拆家1天 vs 拆家3个月，严重度应不同）
- 未考虑行为频率（每天咬人 vs 每月咬人1次）
- 未结合品种特性（哈士奇拆家 vs 法斗拆家，严重度不同）

#### B-04: 训练建议模板化 🟡
**现状代码**: `behavior_tools.py:490-557`
```python
TRAINING_PLANS: ClassVar[Dict[str, Any]] = {
    "destructive": {"methods": [...], "tips": [...]},
    # 仅5类
}
```
**问题**:
- 训练建议完全硬编码，无个性化
- 未根据LLM分析的原因匹配对应训练方案
- 未考虑宠物年龄、性格、主人经验水平
- 训练计划无时间线、无进度跟踪机制

## 三、优化方案

### 3.1 症状分析优化方案

#### OPT-H-01: 紧急度评估升级（多因素评分模型）

**技术方案**:
```
当前: 关键词匹配 → 返回固定级别
优化后: 多因素评分系统

紧急度评分 = BaseScore(关键词) + SpeciesModifier + AgeModifier + HistoryModifier

BaseScore: 
  - 症状关键词匹配得分 (0-3分)
  - LLM诊断severity映射 (轻微=0, 中等=1, 严重=2, 紧急=3)
  - 症状数量叠加 (多个症状+1分)

SpeciesModifier:
  - 猫: 拒食+1, 呕吐+1 (猫代谢更快)
  - 狗: 基础值
  
AgeModifier:
  - < 6个月: +1 (幼宠脆弱)
  - > 96个月: +1 (老年宠物)

HistoryModifier:
  - 有相同症状历史: +1 (可能恶化)
  - 有相关慢性疾病: +1

最终: score 0-1 → level 1, 2-3 → level 2, 4-5 → level 3, 6-7 → level 4, 8+ → level 5
```

**实现复杂度**: 🟡 中 (需修改urgency评估方法，增加评分逻辑)

**资源需求**:
- 新增: 物种差异配置表、年龄风险系数表
- 修改: `_assess_urgency` 方法 (约150行新增代码)

**预期指标**:
- 紧急症状识别率: 60% → 95%
- 误报率: 15% → 5%
- 处理时间: 无增加

**风险评估**:
- 🔵 低风险: 核心逻辑改动，不影响现有接口
- 注意: 需确保评分权重合理，避免过度敏感

#### OPT-H-02: 混合检索策略 (Hybrid Search)

**技术方案**:
```
当前: 向量检索(n=5) → 返回结果
优化后: BM25 + 向量混合检索

Step 1: BM25关键词检索 (精确匹配医学术语)
Step 2: 向量相似度检索 (语义匹配)
Step 3: RRF (Reciprocal Rank Fusion) 合并排序
Step 4: 相关度阈值过滤 (< 0.3 丢弃)
Step 5: 动态调整n值 (症状多→n=8, 症状少→n=3)
Step 6: 去重 (相同知识点仅保留最高分)
```

**实现复杂度**: 🟡 中 (需集成ChromaDB的hybrid搜索或自实现RRF)

**资源需求**:
- 新增: BM25索引构建模块 (~200行)
- 修改: `vector_db.query` 方法调用方式
- 依赖: `rank_bm25` 或 `whoosh` 库

**预期指标**:
- 检索相关度: +30% (RRF融合优势)
- 低质量结果过滤: 减少70%无效知识传入LLM
- LLM token消耗: 减少40% (更精准的top-k)

**风险评估**:
- 🟡 中风险: 需测试BM25分词效果（中文需要jieba）
- 注意: 知识库需预先建立BM25索引（一次性操作）

#### OPT-H-03: 结构化LLM输出

**技术方案**:
```
当前: StrOutputParser + 手动json.loads
优化后: Pydantic schema + JsonOutputParser + 自动重试

from langchain_core.output_parsers import JsonOutputParser
from pydantic import BaseModel, Field

class SymptomAnalysis(BaseModel):
    possible_conditions: List[Condition]
    recommendations: List[str]
    severity: Literal["轻微", "中等", "严重", "紧急"]
    vet_recommended: bool

class Condition(BaseModel):
    name: str
    description: str
    confidence: float = Field(ge=0, le=1)

chain = prompt | llm.with_structured_output(SymptomAnalysis)
```

**实现复杂度**: 🟢 低

**资源需求**:
- 修改: `analyze_with_llm` 方法 (约50行改动)
- 新增: Pydantic schema定义 (~40行)
- 依赖: `langchain-core` 已支持 `with_structured_output`

**预期指标**:
- JSON解析失败率: 20% → < 2%
- 输出格式一致性: 显著提升
- LLM输出质量控制: 通过schema自动约束

**风险评估**:
- 🔵 低风险: LangChain原生功能，兼容GLM
- 注意: 需验证GLM模型是否支持 `response_format={"type": "json_object"}`

#### OPT-H-04: 健康记录时间序列分析

**技术方案**:
```
当前: 简单文本拼接
优化后: 结构化时间序列分析

1. 提取时间线: 按record_date排序
2. 症状模式识别:
   - 相同症状重复出现 → 慢性问题标记
   - 症状频率增加 → 恶化趋势标记
   - 新症状与旧症状关联 → 并发症预警
3. 生成结构化摘要:
   {
     "chronic_issues": ["关节炎(2024-2025反复3次)"],
     "recent_changes": ["近1月新增呕吐症状"],
     "risk_factors": ["糖尿病史+当前多饮多尿 → 血糖失控风险"],
   }
```

**实现复杂度**: 🟡 中

**资源需求**:
- 新增: `HealthRecordAnalyzer` 类 (~150行)
- 修改: `HealthConsultTool._run` 集成分析结果

**预期指标**:
- 历史病历利用率: 提升3倍
- 诊断准确性: +25% (基于历史趋势)

**风险评估**:
- 🟡 中风险: 需确保健康记录数据质量
- 注意: 隐私保护，分析结果不存储敏感信息

#### OPT-B-03: 严重程度评估升级

**技术方案**:
```
当前: 关键词匹配 → 返回1-4
优化后: 多因素评分 + 时间/频率感知

严重程度评分 = 
  BehaviorTypeScore(行为类别基础分) +
  DurationScore(持续时间) +
  FrequencyScore(发生频率) +
  BreedModifier(品种修正) +
  ImpactScore(对人/环境影响)

BehaviorTypeScore:
  - 攻击行为: 3 (基础分最高)
  - 自我伤害: 3
  - 破坏行为: 2
  - 噪音行为: 1
  - 其他: 1

DurationScore:
  - < 1周: 0
  - 1-4周: +1
  - 1-3月: +2
  - > 3月: +3

FrequencyScore:
  - 偶尔(<1次/周): 0
  - 经常(1-3次/周): +1
  - 频繁(>3次/周): +2
  - 持续: +3

最终: score 0-2 → level 1, 3-4 → level 2, 5-6 → level 3, 7-8 → level 4, 9+ → level 5
```

**实现复杂度**: 🟢 低

**资源需求**:
- 修改: `_assess_severity` 方法 (约100行)
- 新增: 用户输入中解析时间/频率信息 (正则提取)

**预期指标**:
- 严重度评估准确率: +40%
- 区分度: 从4级扩展到5级

**风险评估**:
- 🔵 低风险: 纯算法改进
- 注意: 需从用户描述中解析时间信息（如"已经拆家一个月了"）
