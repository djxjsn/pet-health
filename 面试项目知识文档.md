### （四）原理逻辑

#### 1. 知识点：JWT认证机制

**原理**：JWT由三部分组成（Base64编码的JSON）：
- **Header**：包含算法类型（HS256）和令牌类型（JWT）
- **Payload**：包含声明（claims），如user_id、exp（过期时间）、type（令牌类型）
- **Signature**：使用HMAC-SHA256对前两部分签名，确保数据完整性

**工作流程**：
```
用户登录 → 服务器验证凭据 → 生成JWT → 返回给客户端
客户端请求 → 携带JWT → 服务器验证签名 → 确认身份 → 处理请求
```

**项目应用**：
```python
# Token Payload结构
{
    "sub": "user_id",      # 用户标识
    "type": "access",      # 令牌类型
    "exp": 1712600000      # 过期时间
}

# 令牌校验流程
def verify_token(token: str) -> User:
    payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
    if payload["type"] != "access":
        raise HTTPException(401, "Invalid token type")
    return get_user_by_id(payload["sub"])
```

#### 2. 知识点：bcrypt密码哈希

**原理**：
- 加盐：每个密码有唯一的盐值，防止彩虹表
- 迭代：2^12次Blowfish加密，计算成本高
- 自适应：可通过增加rounds应对硬件提升

**项目应用**：
```python
# 密码哈希
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))

# 密码验证
is_valid = bcrypt.checkpw(password.encode(), hashed)
```

### （四）原理逻辑

#### 1. 知识点：FastAPI依赖注入系统

**原理**：FastAPI的依赖注入基于Python的类型注解和`Depends()`函数。当请求到达时，FastAPI按依赖链从最底层开始解析，将结果注入到端点函数参数中。

**依赖链示例**：
```python
# 最底层依赖
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

# 中间层依赖（依赖get_db）
def get_current_user(db: Session = Depends(get_db), token: str = Depends(oauth2_scheme)):
    # 验证token并返回用户
    user = verify_token(token)
    return user

# 端点依赖（依赖get_current_user）
@router.get("/pets/{pet_id}")
def get_pet(
    pet_id: str,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user)  # 自动注入
):
    pet = crud.get_pet_by_id(pet_id)
    if pet.user_id != current_user.user_id:
        raise HTTPException(403)  # 权限校验
    return pet
```

#### 2. 知识点：SQLAlchemy ORM关系映射

**原理**：
- `relationship()`在ORM层建立Python对象间的导航属性
- `back_populates`实现双向引用
- `ForeignKey`在数据库层建立外键约束
- `cascade="all, delete-orphan"`确保父对象删除时子对象自动删除

**项目应用**：
```python
# User模型
class User(Base):
    __tablename__ = "users"
    user_id = Column(String(36), primary_key=True)
    pets = relationship("Pet", back_populates="user", cascade="all, delete-orphan")

# Pet模型
class Pet(Base):
    __tablename__ = "pets"
    pet_id = Column(String(36), primary_key=True)
    user_id = Column(String(36), ForeignKey("users.user_id", ondelete="CASCADE"))
    user = relationship("User", back_populates="pets")
```

#### 3. 知识点：Pydantic v2数据验证

**原理**：
- Rust编写的核心（pydantic-core）实现高性能数据验证
- `@field_validator`在字段级别添加自定义验证逻辑
- `model_dump(exclude_unset=True)`只导出用户显式设置的字段，实现PATCH语义

**项目应用**：
```python
class PetUpdate(BaseModel):
    name: Optional[str] = None
    species: Optional[str] = None
    weight: Optional[float] = None

    @field_validator('species')
    @classmethod
    def validate_species(cls, v):
        if v is None:
            return v
        valid_species = ['dog', 'cat', 'bird', 'rabbit', 'hamster', 'fish', 'other']
        if v.lower() not in valid_species:
            raise ValueError(f'species must be one of {valid_species}')
        return v.lower()  # 自动转小写
```

### （四）原理逻辑

#### 1. 知识点：Zustand 状态管理原理

**原理**：Zustand 基于 React 的 useSyncExternalStore Hook 实现。核心流程：
1. `create()` 创建一个 Store 对象，内部维护状态和订阅者列表
2. 组件通过 `useStore(selector)` 订阅 Store 的部分状态
3. 状态变更时，Zustand 使用浅比较判断 selector 返回值是否变化
4. 仅变化时触发组件重渲染，实现精确更新

**与 Redux 对比**：
- 无需 Provider 包裹（Redux 需要 `<Provider store>`）
- 无需 reducer/action 定义（Zustand 直接 `set()` 修改）
- 无需 useDispatch/useSelector（Zustand 直接调用 Store 方法）
- persist 中间件替代 redux-persist，配置更简洁

**项目应用**：
```typescript
const useAuthStore = create<AuthState>()(
  persist(
    (set) => ({
      user: null,
      token: null,
      isAuthenticated: false,
      login: (user, token) => set({ user, token, isAuthenticated: true }),
      logout: () => set({ user: null, token: null, isAuthenticated: false }),
    }),
    { name: 'auth-storage', partialize: (state) => ({ user: state.user, token: state.token }) }
  )
);
```

#### 2. 知识点：Next.js Middleware 工作原理

**原理**：Next.js Middleware 运行在 Edge Runtime（类似 Cloudflare Workers），在请求到达页面组件之前执行。核心流程：
1. 用户发起请求 → Next.js 路由匹配
2. 如果路径匹配 `matcher` 规则，先执行 middleware 函数
3. middleware 可以读取请求信息（Cookie、Header、URL）
4. 返回 `NextResponse.redirect()` 重定向，或 `NextResponse.next()` 放行

**为什么需要双层守卫**：
- **middleware（服务端）**：在 HTML 传输前拦截，避免未授权内容闪烁，但只能读 Cookie
- **AuthGuard（客户端）**：可以读 Zustand 状态，处理更细粒度的权限逻辑

**项目应用**：
```typescript
export function middleware(request: NextRequest) {
  const token = request.cookies.get('auth-token')?.value;
  const isPublicPath = PUBLIC_PATHS.some(p => pathname.startsWith(p));

  if (!isPublicPath && !token) {
    return NextResponse.redirect(new URL('/login', request.url));
  }
  if (isPublicPath && token) {
    return NextResponse.redirect(new URL('/', request.url));
  }
  return NextResponse.next();
}
```

#### 3. 知识点：Toast 通知生命周期管理

**原理**：Toast 系统采用"添加时启动定时器"模式，而非"组件内 useEffect 定时"模式。核心流程：
1. `addToast()` 创建 Toast 对象，分配唯一 ID
2. 同时启动 `setTimeout` 定时器，到期后从 store 中移除
3. 组件渲染时，在 duration 结束前 300ms 触发退出动画
4. 手动关闭时，先触发退出动画，300ms 后再从 store 移除

**为什么不在组件 useEffect 中管理定时器**：
- 组件可能因父组件重渲染而卸载/重新挂载，导致定时器丢失
- Store 层管理定时器更稳定，与组件生命周期解耦

**项目应用**：
```typescript
addToast: (toast) => {
  const id = `toast-${++toastCounter}-${Date.now()}`;
  const duration = toast.duration ?? (toast.type === 'error' ? 6000 : 4000);
  set((state) => ({ toasts: [...state.toasts, { ...toast, id, duration }] }));
  if (duration > 0) {
    setTimeout(() => {
      set((state) => ({ toasts: state.toasts.filter((t) => t.id !== id) }));
    }, duration);
  }
  return id;
}
```

#### 2. 知识点：React.memo 工作原理与适用场景

**原理**：
```
普通组件:
  Parent re-render → Child always re-render (无论 props 是否变化)

memo 组件:
  Parent re-render → Child shallow compare props
    ├── props 相同 → 跳过渲染 (return cached result)
    └── props 不同 → 执行重新渲染
```

**浅比较 (Shallow Compare)** 规则：
- 基本类型：值相等 (`a === b`)
- 对象/数组：引用相等 (`a === b`，不是深度比较)
- 函数：引用相等（每次 render 创建的新函数 ≠ 上次的）

**项目中的应用场景判断**：

| 组件 | 为什么适合 memo？ | 注意事项 |
|------|------------------|---------|
| MessageBubble | props 只有 message 对象，消息列表中频繁出现 | message 引用需稳定（useMemo 包装） |
| PetSwitcher | pets 数组可能频繁更新但当前选中不变 | currentPetId 未变时不重渲染 |
| Spinner | 纯展示组件，props 仅 size/className | 无状态无副作用，memo 收益最大 |

**不适合 memo 的场景**：
- 组件内部有 hooks 依赖外部状态变化
- props 经常变化（如每次 render 新建对象）
- 组件本身很轻量（memo 开销 > 重渲染开销）

  try {
    const response = await fetchWithTimeout(url, options, timeout);
    if (!response.ok) {
      if (response.status < 500) throw parseErrorResponse(response.status, body);
      lastError = error;
      await new Promise(r => setTimeout(r, 1000 * Math.pow(1.5, attempt)));
      continue; // 仅 5xx 重试
    }
    return await response.json();
  } catch (error) {
    lastError = error;
    if (attempt < retries) {
      await new Promise(r => setTimeout(r, 1000 * Math.pow(1.5, attempt)));
      continue;
    }
  }

#### 3. 知识点：前端错误处理的分层架构

│     statusCode?: number

├── handleAppError(error, context?)
│   ├── console.error (开发者日志)

#### 1. 数据库Schema不一致问题排查

**问题现象**：
```
sqlalchemy.exc.OperationalError: (pymysql.err.OperationalError) (1054, "Unknown column 'user_profiles_1.pet_count' in 'field list'")
```

**排查思路**：
1. 错误信息指向 `user_profiles` 表缺少 `pet_count` 列
2. 检查 SQLAlchemy 模型 `UserProfile` — 定义了 `pet_count` 字段
3. 检查 Alembic 迁移脚本 `001_create_users_and_profiles.py` — 发现迁移脚本中未包含此列
4. 根因：Alembic 迁移脚本与 SQLAlchemy 模型不同步（开发过程中模型变更但迁移脚本未更新）

**解决方案**：
```sql
ALTER TABLE user_profiles ADD COLUMN pet_count INT NOT NULL DEFAULT 0 COMMENT 'pet count' AFTER avatar_url;
```

**经验沉淀**：模型变更后必须同步更新 Alembic 迁移脚本，或在团队中建立 Schema 变更 review 机制。

#### 2. OAuth2 FormData 登录格式

**问题现象**：前端 `authApi.login()` 始终返回 400 错误

**排查思路**：
1. 检查后端 `auth.py` — 使用 FastAPI OAuth2PasswordRequestForm 依赖
2. OAuth2PasswordRequestForm 要求 `application/x-www-form-urlencoded` 格式
3. 前端最初使用 JSON 格式发送 `{ username, password }`

**解决方案**：
```typescript
// 错误方式
fetch(url, { method: 'POST', body: JSON.stringify({ username, password }) })

// 正确方式
const formData = new FormData();
formData.append('username', username);
formData.append('password', password);
fetch(url, { method: 'POST', body: formData });
// 或者使用 application/x-www-form-urlencoded
fetch(url, { method: 'POST', body: 'username=xxx&password=yyy', headers: { 'Content-Type': 'application/x-www-form-urlencoded' } })
```

#### 3. LLM服务不可用的容错设计

**问题现象**：Chat/Health/Behavior 端点请求超时（HuggingFace 不可达）

**容错验证结果**：
- ✅ API层正确超时分类（TIMEOUT ErrorCategory）
- ✅ 前端 Toast 正确展示超时提示
- ✅ 非LLM端点（CRUD/认证）完全正常

**经验沉淀**：外部服务依赖必须设置合理的超时时间，并在UI层提供降级体验。

#### 4. WebSocket 热重载问题

**问题现象**：Uvicorn `--reload` 模式下 WatchFiles 检测到测试脚本变更后崩溃

**解决方案**：生产/CI环境使用非 reload 模式：`uvicorn src.main:app --host 0.0.0.0 --port 8000`

## OPT-H-03/OPT-B-03 - LLM结构化输出优化

【总结标注】项目名称：AI宠物健康助手  具体任务/阶段性工作名称：OPT-H-03/OPT-B-03 LLM结构化输出优化  完成日期：2026-04-16

### （一）项目解决问题的过程

#### 1. 具体问题
项目中健康咨询和行为分析模块的LLM输出完全依赖prompt中的文字描述来要求返回JSON格式，实际运行中LLM经常返回非标准JSON（包含额外文字、格式错误、字段缺失等），导致`json.loads()`解析失败率高达20%，只能降级到简陋的fallback方案。

#### 2. 分析过程
- **根因分析**：`StrOutputParser`仅将LLM输出转为字符串，无任何格式约束
- **影响范围**：`health_tools.py`症状分析 + `behavior_tools.py`行为分析
- **降级方案问题**：降级后仅返回知识库前3条，丧失LLM推理能力
- **解决方案调研**：LangChain提供`JsonOutputParser` + Pydantic schema实现强制结构化输出

#### 3. 解决方案及结果
- **新建Schema层**：`src/schemas/llm_output.py`定义6个Pydantic模型
  - `ConditionItem`: 症状分析中的疾病项（name, description, confidence 0-1）
  - `SymptomAnalysisResult`: 症状分析完整结果
  - `BehaviorCauseItem`: 行为分析中的原因项（cause, probability 0-1, category）
  - `BehaviorAnalysisResult`: 行为分析完整结果
  - `HealthConsultationOutput`: 健康咨询完整输出
  - `BehaviorAnalysisOutput`: 行为分析完整输出
- **health_tools.py改造**：
  - 移除`StrOutputParser`，改用`JsonOutputParser(pydantic_object=SymptomAnalysisResult)`
  - Prompt中注入`{format_instructions}`（Pydantic自动生成的JSON schema描述）
  - 移除手动`json.loads()`，Parser直接返回Dict
- **behavior_tools.py改造**：
  - 同上模式，使用`BehaviorAnalysisResult`
- **前端类型对齐**：`frontend/src/types/api.ts`新增对应TypeScript接口
- **后端API Schema更新**：`schemas/health.py`和`schemas/behavior.py`引用新的结构化类型

#### 4. 经验教训
- LLM输出必须强制约束，不能仅靠prompt文字描述
- Pydantic schema同时服务于后端验证和前端类型生成
- Parser异常需保留fallback，但fallback应尽可能接近正常输出

### （二）项目技术难点与亮点

#### 1. 技术难点
- **JSON解析失败**：20%的LLM输出格式不符合预期，手动解析不可靠
- **字段类型约束**：confidence必须是0-1浮点数，severity必须是四个字面量之一
- **前后端类型同步**：Pydantic schema需要对应到TypeScript接口

#### 2. 技术亮点
- **Pydantic + JsonOutputParser组合**：LangChain原生能力，零额外依赖
- **format_instructions自动注入**：无需手动编写JSON schema描述
- **类型安全全链路**：Pydantic → FastAPI Response → TypeScript Interface 一致
- **优雅降级**：Parser异常时fallback到知识库基础分析

### （三）涉及的知识点

1. **LangChain领域**：
   - `JsonOutputParser`: 强制LLM输出标准JSON格式
   - `pydantic_object`参数：传入Pydantic模型实现字段类型约束
   - `get_format_instructions()`: 自动生成JSON格式说明注入到prompt中
   - `StrOutputParser` vs `JsonOutputParser`: 前者纯字符串，后者解析为Dict/List

2. **Pydantic领域**：
   - `Field(ge=0, le=1)`: 数值范围约束
   - `Literal["轻微", "中等", "严重", "紧急"]`: 枚举字面量约束
   - `model_dump()`: Pydantic模型转字典（FastAPI自动调用）

3. **FastAPI领域**：
   - `response_model`: 自动验证和序列化响应数据
   - 嵌套Pydantic模型自动序列化

### （四）原理逻辑

1. **知识点 JsonOutputParser 工作原理**：
   - **原理**：解析器在chain末端接收LLM输出文本 → 尝试提取JSON（支持markdown代码块包裹） → 用Pydantic schema验证字段类型和必填项 → 验证通过返回Dict，失败抛异常
   - **应用体现**：症状分析中LLM可能返回"根据分析...\n```json\n{...}\n```"，Parser自动提取JSON部分并验证

2. **知识点 format_instructions 生成机制**：
   - **原理**：Pydantic模型的`model_json_schema()`生成JSON Schema → Parser将其格式化为人类可读的文本说明 → 注入到prompt的`{format_instructions}`占位符
   - **应用体现**：自动生成的说明类似"输出应为JSON对象，包含possible_conditions(数组)、severity(枚举)、vet_recommended(布尔)等字段"

3. **知识点 全链路类型安全**：
   - **原理**：Python Pydantic schema → FastAPI自动序列化 → JSON → TypeScript interface → 前端类型检查
   - **应用体现**：后端修改SymptomAnalysisResult增加字段，前端TypeScript编译立即报错，确保一致性

## ARCH-001 - MySQL → MongoDB 混合架构迁移

**任务标注**：项目名称：AI宠物健康助手  具体任务/阶段性工作名称：MySQL → MongoDB 混合架构迁移  完成日期：2026-04-16

### （一）项目解决问题的过程

1. **具体问题**：
   - 所有数据存储在 MySQL，但对话、健康记录、行为分析等数据具有高度非结构化特性（JSON字段多、查询模式简单、无复杂关联）
   - MySQL 文档型数据需要 JSON 序列化/反序列化，增加复杂度
   - 随着数据量增长，MongoDB 的文档模型更适合此类数据

2. **分析过程**：
   - 评估 8 张表的数据模型特征：users/user_profiles/pets 有外键关联、强事务需求 → 保留 MySQL
   - conversations/messages/health_records/consultations/behavior_analyses 无复杂关联、查询简单 → 迁移 MongoDB
   - 设计 Repository 模式抽象数据库操作层，屏蔽 MySQL/MongoDB 差异
   - 使用 UUID 字符串 ID 保持 MySQL 外键与 MongoDB 文档的兼容性

3. **解决方案及结果**：
   - 安装 MongoDB Server v8.2.6（winget），配置连接池（maxPoolSize=50）
   - 创建 `MySQLBase` 独立声明基类，防止 MongoDB 模型自动创建 MySQL 表
   - 编写迁移脚本（备份 → 迁移 → 验证 → 索引），数据完整性 100% 通过
   - 更新 3 个 API 端点使用 MongoDB Repository 替代 SQLAlchemy CRUD
   - 创建 15 个 MongoDB 索引覆盖核心查询场景
   - 集成测试 27 个用例，96.3% 成功率

4. **经验教训**：
   - 环境验证前置：迁移前必须先验证 MongoDB 服务可用性
   - Pydantic 响应模型必须完整构造，不能仅传递字典
   - 测试断言应兼容 200/201 等成功状态码范围
   - Repository 模式有效屏蔽了底层数据库差异

### （二）项目技术难点与亮点

1. **技术难点**：
   - **难点 1：MySQL/MongoDB 模型隔离**
     - SQLAlchemy `declarative_base()` 会自动将所有继承的模型注册到 metadata，导致 `create_all()` 时尝试在 MySQL 中创建 MongoDB 模型对应的表
     - **突破方法**：创建独立的 `MySQLBase = declarative_base()`，仅让 MySQL 模型继承它；MongoDB 模型继承 `BaseModel`（Pydantic）

   - **难点 2：Pydantic 响应模型嵌套构造**
     - `BehaviorAnalyzeResponse` 要求 `analysis_result: BehaviorAnalysisResult`，但 tool 返回的是扁平化字典
     - **突破方法**：在 endpoint 中从 tool 返回结果中提取数据，手动构造嵌套的 Pydantic 模型对象

   - **难点 3：外键约束兼容**
     - MongoDB 无外键约束，messages 表的 `conversation_id` 等关联字段需应用层保证
     - **突破方法**：使用 UUID 字符串 ID 作为文档 `_id`，保持与 MySQL 外键格式一致；Repository 层封装关联查询逻辑

2. **技术亮点**：
   - **Repository 模式**：统一数据库操作接口，未来切换数据库无需修改业务代码
   - **单例连接管理**：`MongoDBClient` 使用单例模式，全局复用连接，避免重复连接开销
   - **迁移脚本完备**：包含连接检查 → 数据备份 → 数据迁移 → 完整性验证 → 索引创建全流程
   - **零停机迁移**：先写入 MongoDB，验证无误后再切换读取源，保证业务连续性

### （三）涉及的知识点

1. **数据库领域**：
   - **MySQL vs MongoDB 选型**：关系型数据（外键、事务）用 MySQL；文档型数据（非结构化、灵活 schema）用 MongoDB
   - **MongoDB 索引策略**：用户 ID 索引（查询用户数据）、时间索引（排序）、状态索引（过滤）
   - **连接池管理**：maxPoolSize=50, minPoolSize=10, maxIdleTimeMS=30000

2. **设计模式领域**：
   - **单例模式**：`MongoDBClient` 确保全局唯一实例
   - **Repository 模式**：封装数据访问逻辑，提供清晰的 CRUD 接口
   - **依赖注入**：FastAPI `Depends()` 获取数据库会话和当前用户

3. **架构领域**：
   - **混合架构**：MySQL + MongoDB 双库并存，各取所长
   - **分层架构**：API 层 → Repository 层 → 数据库层
   - **数据迁移策略**：备份 → 迁移 → 验证 → 索引 → 切换

### （四）原理逻辑

1. **知识点 SQLAlchemy declarative_base 机制**：
   - **原理**：`declarative_base()` 创建一个元类，所有继承该 Base 的模型类会被自动注册到 `Base.metadata` 中；调用 `Base.metadata.create_all(engine)` 时会遍历 metadata 中的所有模型并在数据库中创建对应的表
   - **应用体现**：项目中创建 `MySQLBase` 作为 MySQL 专用 Base，仅 users/user_profiles/pets 继承它；MongoDB 模型继承 Pydantic `BaseModel` 不注册到 SQLAlchemy metadata

2. **知识点 MongoDB 文档模型**：
   - **原理**：MongoDB 使用 BSON 格式存储文档，每个文档是一个类似 JSON 的对象；集合（collection）是无 schema 的，但应用层可使用 Pydantic 进行数据验证
   - **应用体现**：使用 `Pydantic` 模型定义文档结构，`model_dump(mode="json")` 将模型转换为可存储的 dict

3. **知识点 Repository 模式**：
   - **原理**：在数据访问层和应用层之间添加抽象层，Repository 提供统一的 CRUD 接口，隐藏底层数据库实现细节
   - **应用体现**：`ConversationRepository.create(data)` 无论底层是 SQLAlchemy 还是 PyMongo，调用方式不变

4. **知识点 混合架构数据一致性**：
   - **原理**：MySQL 和 MongoDB 之间无分布式事务保证，需应用层确保最终一致性
   - **应用体现**：pets 在 MySQL，conversations 在 MongoDB，通过 `user_id`/`pet_id` UUID 字符串关联；先验证宠物归属（MySQL），再操作对话（MongoDB）
