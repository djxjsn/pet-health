## 二、完成内容

### 1. 数据模型层（Model）

**文件**：`animal/src/models/pet.py`

- 创建 Pet ORM 模型，包含 13 个字段
- 主键使用 UUID（CHAR(36)）
- 外键关联 users 表，级联删除
- Enum 类型字段：gender（male/female/unknown）
- Numeric 类型字段：weight（5,2 精度）
- 时间字段：created_at、updated_at（自动管理）
- 更新 User 模型，添加 pets 一对多关系

### 2. Schema 层（Pydantic）

**文件**：`animal/src/schemas/pet.py`

- PetCreate：创建请求 Schema，包含字段验证
- PetUpdate：更新请求 Schema，所有字段为 Optional
- PetResponse：响应 Schema，支持 ORM 对象转换
- PetListResponse：分页列表响应 Schema
- 自定义验证器：
  - validate_species：限制物种为 dog/cat/bird/rabbit/hamster/fish/other
  - validate_gender：限制性别为 male/female/unknown
  - 支持大小写不敏感输入

### 3. CRUD 层

**文件**：`animal/src/db/crud/pet.py`

- get_pet_by_id：根据 ID 查询宠物
- get_pets_by_user：分页查询用户的所有宠物
- create_pet：创建宠物档案
- update_pet：更新宠物信息
- delete_pet：删除宠物档案
- 所有函数均使用 Session 事务管理

### 4. API 端点层

**文件**：`animal/src/api/v1/endpoints/pets.py`

实现 5 个 RESTful API 接口：

1. **POST /api/v1/pets**
   - 功能：创建宠物档案
   - 状态码：201 Created
   - 权限：需要认证

2. **GET /api/v1/pets**
   - 功能：获取当前用户的所有宠物（分页）
   - 参数：page（默认 1）、page_size（默认 10，最大 100）
   - 状态码：200 OK
   - 权限：需要认证

3. **GET /api/v1/pets/{pet_id}**
   - 功能：获取特定宠物详情
   - 状态码：200 OK
   - 异常处理：404 Not Found（宠物不存在）、403 Forbidden（无权访问）

4. **PUT /api/v1/pets/{pet_id}**
   - 功能：更新宠物信息
   - 状态码：200 OK
   - 权限校验：仅宠物所属用户可更新

5. **DELETE /api/v1/pets/{pet_id}**
   - 功能：删除宠物档案
   - 状态码：204 No Content
   - 权限校验：仅宠物所属用户可删除

## 四、遇到的问题

### 问题 1：测试环境依赖缺失

**现象**：运行 pytest 时报错 ModuleNotFoundError: No module named 'pytest'

**原因**：项目未创建虚拟环境，依赖包未安装

**解决过程**：
1. 安装 pytest 和 pytest-cov
2. 发现缺少 fastapi、sqlalchemy、python-jose、bcrypt 等
3. 逐个安装所有必要依赖
4. 最后安装 httpx、email-validator、python-multipart 等测试和运行时依赖

**经验**：应该在项目初始化时创建 requirements.txt 或 pyproject.toml 管理依赖

### 问题 2：用户数据隔离实现

**现象**：需要确保用户 A 无法访问/修改/删除用户 B 的宠物数据

**原因**：宠物档案是多租户数据，必须进行权限隔离

**解决过程**：
1. 在 API 层通过 `get_current_user` 依赖注入获取当前用户
2. 查询宠物后校验 `pet.user_id == current_user.user_id`
3. 不一致时返回 403 Forbidden
4. 在 CRUD 层保持纯粹的数据操作，不处理权限逻辑

**经验**：权限校验放在 API 层比 CRUD 层更合适，职责分离清晰

### 问题 3：分页查询设计

**现象**：需要同时返回数据列表和总数

**原因**：前端需要总数进行分页显示

**解决过程**：
1. CRUD 层返回 Tuple[List[Pet], int]
2. API 层组装为 PetListResponse，包含 items、total、page、page_size
3. 使用 SQLAlchemy 的 offset/limit 实现分页

**经验**：分页逻辑应在数据库层完成，避免查询所有数据后再切片

### 问题 4：Enum 类型在 SQLite 和 MySQL 的兼容性

**现象**：SQLAlchemy 的 Enum 在不同数据库中行为不一致

**原因**：MySQL 有真正的 ENUM 类型，SQLite 没有

**解决过程**：
1. 在 Model 层使用 SQLAlchemy Enum
2. 在 Schema 层使用 Pydantic 进行严格验证
3. 不依赖数据库级约束，在应用层做验证

**经验**：跨数据库兼容时，应在应用层做验证而非依赖数据库特性

## 五、解决方案总结

### 技术方案

1. **分层架构**：Model → Schema → CRUD → API，职责清晰
2. **依赖注入**：使用 FastAPI 的 Depends 实现认证和数据库会话管理
3. **权限控制**：在 API 层校验 pet.user_id 与 current_user.user_id 一致性
4. **分页查询**：CRUD 层返回 (列表，总数)，API 层组装响应
5. **测试隔离**：SQLite 内存数据库 + dependency_overrides

### 测试方案

1. **Fixture 层级设计**：db_session → client → auth_headers → created_pet
2. **测试分类**：按功能模块分 7 个测试类
3. **覆盖场景**：正常流程、边界条件、权限控制、数据隔离、CRUD 集成
4. **覆盖率工具**：pytest-cov，生成详细覆盖率报告
