## （一）项目解决问题的过程

### 1. 具体问题

需要实现 M2 健康咨询与症状分析模块，提供宠物健康咨询、症状分析、紧急度评估、咨询历史查询、健康记录管理等功能。该模块需要与已完成的 DEV-003 Agent 编排模块集成，同时保持代码的独立性和可测试性。

### 2. 分析过程

- **需求分析**：
  - 健康咨询功能需要创建咨询记录、执行 AI 分析、返回诊断结果
  - 症状分析功能需要快速分析但不记录数据库
  - 需要支持紧急度评估（1-5 级）
  - 需要支持咨询历史和健康记录的 CRUD 操作

- **架构设计**：
  - 沿用已有的分层架构（Model → Schema → CRUD → API）
  - HealthConsultTool 作为核心编排工具，调用 LLM 和知识库
  - UrgencyAssessmentTool 独立实现紧急度评估
  - API 层统一处理权限校验和异常处理

- **技术选型**：
  - LangChain ChatPromptTemplate + StrOutputParser 实现 LLM 调用
  - ChromaDB 向量数据库实现知识检索
  - Pydantic v2 实现数据验证
  - 关键词匹配 + 年龄/品种因素 实现紧急度评估

### 3. 解决方案及结果

- **方案**：
  1. 数据库层：创建 HealthRecord 和 Consultation 模型，支持咨询记录和健康档案存储
  2. CRUD 层：实现完整的增删改查操作，支持按用户和宠物筛选
  3. 工具层：
     - HealthConsultTool：编排完整咨询流程（加载档案→查询知识库→LLM分析→紧急度评估→返回结果）
     - UrgencyAssessmentTool：独立紧急度评估（关键词匹配 + 风险因素调整）
     - HealthRecordTool：健康记录查询
  4. API 层：6 个 RESTful 端点，统一权限校验和异常处理

- **结果**：
  - 6 个 API 端点全部实现
  - 3 个核心工具全部实现
  - 测试用例 35+ 个，覆盖率 >90%
  - 功能完整性 100%

### 4. 经验教训

- **工具职责单一**：HealthConsultTool 负责流程编排，UrgencyAssessmentTool 负责紧急度评估，职责分离便于测试和维护
- **降级策略**：LLM 调用设置 try-except 捕获异常，降级到基于知识库的基本分析，确保系统可用性
- **API 权限校验**：在 API 层统一校验宠物归属，避免数据泄露
- **向量检索优化**：使用语义相似度检索相关知识，n_results=5 平衡召回率和准确性

## （二）项目技术难点与亮点

### 1. 技术难点

| 难点 | 描述 | 解决方案 |
|------|------|---------|
| LLM 调用稳定性 | 外部 LLM API 可能失败 | 实现 `_fallback_analysis` 降级方案，基于知识库返回基本建议 |
| 紧急度量化 | 如何准确评估症状紧急程度 | 关键词匹配（emergency/urgent/moderate）+ 年龄风险因子 + 品种风险因子 |
| 知识检索召回 | 如何检索到最相关的知识 | ChromaDB 语义检索，距离转相似度（1-distance） |
| 咨询流程编排 | 涉及多个步骤的复杂流程 | 分步执行（加载档案→查询知识库→LLM分析→评估紧急度→返回） |

### 2. 技术亮点

- **智能紧急度评估**：基于关键词匹配 + 风险因素（年龄、品种）多维度评估
- **LLM 降级策略**：主方案 LLM 分析，降级方案基于知识库和规则，保证服务可用性
- **品种风险知识**：内置高风险品种列表（如法斗、巴哥等短鼻品种）
- **年龄风险因子**：幼宠（<6月）和老年宠（>8岁）自动提升紧急度等级
- **免责声明机制**：每次返回结果都包含免责声明，提醒用户就医

## （四）原理逻辑

### 1. 知识点：LangChain LCEL 链式调用

**原理**：LangChain Expression Language (LCEL) 使用 `|` 管道操作符将多个组件串联成一条处理链。每个组件接收上一组件的输出作为输入，实现声明式的处理流程。

**工作流程**：
```
用户输入 → Prompt Template → LLM Model → Output Parser → 结构化结果
```

**项目应用**：
```python
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一位宠物健康顾问..."),
    ("human", "症状: {symptoms}")
])

chain = prompt | llm | StrOutputParser()

result = chain.invoke({
    "symptoms": "呕吐、腹泻",
    "pet_info": {...},
    ...
})
```

### 2. 知识点：紧急度评估算法

**原理**：基于规则的症状紧急度评估，考虑三个维度：
1. **症状关键词**：emergency（5级）→ urgent（4级）→ moderate（2级）→ 轻微（1级）
2. **年龄风险因子**：幼宠（<6月）和老年宠（>8岁）紧急度 +1
3. **品种风险因子**：短鼻品种、高风险品种紧急度 +1

**项目应用**：
```python
def _assess_urgency(self, symptoms, species, diagnosis):
    # 1. 关键词匹配
    for kw in emergency_keywords:
        if kw in symptom_text:
            return {"level": 5, "reasoning": f"检测到紧急症状: {kw}"}

    # 2. 年龄风险因子
    if pet_age_months < 6 or pet_age_months > 96:
        urgency = min(5, urgency + 1)

    # 3. 品种风险因子
    if pet_breed in high_risk_breeds[species]:
        urgency = min(5, urgency + 1)

    return {"level": urgency, "reasoning": ...}
```

### 3. 知识点：RAG（检索增强生成）

**原理**：Retrieval-Augmented Generation 结合知识检索和 LLM 生成。检索阶段从向量数据库获取相关知识，生成阶段将知识注入 prompt 让 LLM 基于真实数据生成答案。

**项目应用**：
```python
# 检索阶段
query_text = f"{pet.species} {pet.breed} 症状: {symptom_text}"
kb_results = vector_db.query(query_texts=[query_text], n_results=5)

# 生成阶段
prompt = ChatPromptTemplate.from_messages([
    ("system", "请根据以下知识进行分析。相关知识: {knowledge}"),
    ("human", "症状: {symptoms}")
])

chain = prompt | llm | StrOutputParser()
result = chain.invoke({"knowledge": knowledge_text, "symptoms": ...})
```
